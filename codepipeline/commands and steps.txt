üîπ Fundamentals
Q1. What is AWS CodePipeline?

A fully managed CI/CD orchestration service that automates code release workflows across AWS services.

Q2. Why did you introduce CodePipeline when CodeDeploy already existed?

CodeDeploy handles deployment, but CodePipeline automates the entire release flow, removing manual CLI/console triggers.

Q3. What problem does CodePipeline solve in this lab?

It enables automatic deployments whenever application code changes in the source bucket.

üîπ High-Level Architecture (VERY IMPORTANT)
Q4. Explain the architecture of this lab.

CodePipeline monitors an S3 bucket, detects changes, and triggers CodeDeploy, which deploys the application to EC2 instances across multiple AZs.

Q5. What are the stages used in this pipeline?

Source: Amazon S3

Deploy: AWS CodeDeploy
(Build and Test stages were skipped)

Q6. Why is S3 used as the source stage?

Because the application artifacts are pre-built ZIP files, not source code requiring compilation.

üîπ Source Stage (S3)
Q7. Why must S3 versioning be enabled for CodePipeline?

CodePipeline tracks object versions to detect changes and trigger deployments.

Q8. How does CodePipeline detect a change in S3?

Using Amazon EventBridge (CloudWatch Events) on object upload.

Q9. What triggers the pipeline in this lab?

Uploading or overwriting HeartBeat-App.zip in the S3 source bucket.

Q10. Is polling used here?

No. The pipeline uses event-based triggers, not polling.

üîπ IAM & Security
Q11. Why does CodePipeline need a service role?

To interact with S3, CodeDeploy, CloudWatch, and other AWS services securely.

Q12. What role was used in this lab?

AWSCodePipelineServiceRole-<region>-HeartBeatPipeline

Q13. Principle followed by the service role?

Least privilege ‚Äî permissions limited to required services only.

üîπ Build & Test Stages (Tricky Interview Area)
Q14. Why was the build stage skipped?

The application was already compiled and packaged as a ZIP.

Q15. When would you need a build stage?

When source code needs compilation, testing, or artifact creation (using CodeBuild).

Q16. Can you add build/test stages later?

Yes, pipelines are fully extensible.

üîπ Deploy Stage (CodeDeploy)
Q17. Why was CodeDeploy used as the deploy provider?

Because it supports Windows services, lifecycle hooks, and EC2 fleet deployments.

Q18. What deployment type was used?

In-place deployment on existing EC2 instances.

Q19. How does CodePipeline know where to deploy?

It references the CodeDeploy application and deployment group.

Q20. How were target instances selected?

Using EC2 tags (Name = HeartBeat-Instance).

üîπ Automation & Flow Control
Q21. What happens after uploading a new ZIP to S3?

Source stage succeeds

Deploy stage calls CodeDeploy

CodeDeploy runs appspec hooks

Q22. Is any manual action required after pipeline creation?

No. Deployments are fully automatic.

Q23. How can you manually force a deployment?

Using Release change in the CodePipeline console.

üîπ Monitoring & Observability

Q26. What happens if deployment fails?

Pipeline shows failed state, and rollback can occur if configured.
